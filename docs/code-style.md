# Code-style

Документ фиксирует базовые правила разработки Klinomania. Если правило не описано здесь, используем стандартные соглашения Laravel.

## Базовый стиль

- Следуем Laravel conventions для именования классов, контроллеров, моделей, миграций, фабрик и тестов.
- Форматирование PHP-кода выполняется через Laravel Pint.
- Тесты пишутся на Pest: feature-тесты для пользовательских сценариев и unit-тесты для изолированной бизнес-логики.
- Новые классы должны иметь явные типы аргументов и возвращаемых значений там, где это не конфликтует с Laravel и Eloquent.
- Предпочитаем понятные имена доменных методов вместо технических сокращений.

## Контроллеры и бизнес-логика

- Контроллеры принимают HTTP-запрос, вызывают прикладное действие или сервис и возвращают ответ.
- Бизнес-логику не размещаем в контроллерах.
- Сложные сценарии оформляем как action-классы внутри соответствующего модуля.
- Модели Eloquent отвечают за состояние и связи, но не должны напрямую управлять внешними API или сложными пользовательскими сценариями.

## Интеграции и конфигурация

- Внешние API вызываем через отдельные клиенты, gateway или сервисы.
- Интеграция с Т-Банком должна быть скрыта за доменным контрактом платежного провайдера.
- Значения из окружения читаем через config-файлы. В прикладном коде не используем `env()` напрямую.
- Секреты, ключи API и URL внешних сервисов хранятся в `.env`, а в код попадают через `config()`.

## Тестирование

- Критические сценарии бронирования и оплаты покрываем feature-тестами.
- Расчет суммы заказа, переходы статусов и обработку платежных статусов покрываем unit- или feature-тестами в зависимости от уровня интеграции.
- В тестах не обращаемся к реальным API Т-Банка. Используем fake-реализации gateway или HTTP fake.

## Документация

- Архитектурные решения и правила проекта фиксируются в `docs/`.
- Root `README.md` не используется как основное место проектной документации.
- Если меняется архитектурное правило, обновляем соответствующий документ вместе с изменением кода.
