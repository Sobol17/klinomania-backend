# Архитектура

Klinomania - backend для мобильного приложения клининга и административной панели. Приложение строится как модульный монолит на Laravel 13: один деплой и одна база данных, но доменная логика разделяется по модулям.

## Архитектурный стиль

Модульный монолит выбран как баланс между скоростью разработки и контролем сложности. Каждый модуль отвечает за свой домен и содержит связанные модели, HTTP-слой, action-классы, DTO/data-классы и интеграции. Общий код выносится в `Shared`.

Контроллеры остаются тонкими. Основные пользовательские сценарии реализуются в action-классах. Интеграции с внешними сервисами изолируются за контрактами, чтобы прикладная логика не зависела от конкретного API.

## Поверхности приложения

- Mobile API - JSON API для клиентского и клинерского мобильных приложений.
- Filament admin - веб-админка для управления пользователями, клинерами, услугами, заявками и операционными действиями.
- Queue workers - фоновые задачи для уведомлений, интеграций и будущих платежных callbacks.

API версионируется через префикс `/api/v1`.

## Роли

- `client` - клиент мобильного приложения, создает заявки на уборку и смотрит историю.
- `cleaner` - исполнитель, видит доступные заявки, берет заявку в работу и завершает ее.
- `admin` - пользователь Filament admin.

Проверки доступа выполняются на уровне маршрутов, policies и прикладных действий. Filament доступен только администраторам.

## Авторизация

### Клиент

Клиентская авторизация состоит из двух этапов:

1. Клиент отправляет номер телефона на `POST /api/v1/client/auth/request-code`.
2. Сервис генерирует одноразовый 4-значный код, сохраняет hash кода и отправляет SMS.
3. Клиент отправляет телефон и код на `POST /api/v1/client/auth/verify-code`.
4. После успешной проверки создается или находится пользователь с ролью `client` и выдается Sanctum token.

Пользователь клиента создается только после успешного `verify-code`. Неподтвержденные номера не попадают в `users`.

OTP хранится в таблице auth-кодов с коротким временем жизни, счетчиком попыток и отметкой использования. Коды не логируются в открытом виде и не хранятся как plain text.

### Клинер

Клинер создается в админке. Администратор генерирует 6-значный код доступа. Клинер входит по номеру телефона и этому коду через `POST /api/v1/cleaner/auth/login`. Код хранится только в hash-виде. После успешной проверки выдается Sanctum token.

### Admin

Администратор входит через Filament web-auth. Доступ разрешен только пользователям с ролью `admin`.

## Интеграции

### Notisend

SMS отправляются через контракт `SmsGateway`. Первая production-реализация должна использовать Notisend. Конфигурация хранится в `config/services.php`, секреты - в `.env`.

В тестах и локальной разработке используется fake/log gateway без реальных SMS-запросов. Прикладной код не должен зависеть от Notisend напрямую.

## Целевые модули

- `Identity` - роли, Sanctum tokens, клиентский SMS OTP flow, cleaner login code, SMS gateway.
- `Services` - каталог услуг клининга, базовые цены, активность услуг.
- `Checkout` - оформление заявки, расчет итоговой стоимости, подготовка заказа к оплате.
- `Orders` - заявки на уборку, статусы, клиентская история.
- `CleanerWork` - листинг доступных заявок, принятие заявки, старт и завершение работы.
- `Profiles` - профили клиента и клинера.
- `Admin` - Filament resources и административные actions.
- `Payments` - платежные намерения, callbacks/webhooks, статусы оплаты. Добавляется при подключении оплаты.
- `Notifications` - SMS, push/email-уведомления и шаблоны сообщений.
- `Shared` - общие enum, value objects, исключения, базовые контракты и support-классы.

## Основной бизнес-процесс

1. Клиент авторизуется по номеру телефона и SMS-коду.
2. Клиент выбирает услугу из каталога.
3. Клиент оформляет заявку: адрес, желаемое время, услуга, комментарий.
4. Заявка становится доступной клинерам.
5. Клинер авторизуется по телефону и 6-значному коду.
6. Клинер видит доступные заявки и берет одну к исполнению.
7. Клинер приступает к работе.
8. Клинер завершает заявку.
9. Клиент и клинер видят заявку в истории.

## Статусы заявки

- `pending` - заявка создана и ожидает клинера.
- `accepted` - клинер взял заявку.
- `in_progress` - клинер приступил к работе.
- `completed` - уборка завершена.
- `cancelled` - заявка отменена.

Переходы статусов должны выполняться через прикладные actions, а не прямым изменением из контроллеров.

## Базовые API endpoints

- `POST /api/v1/client/auth/request-code` - запросить 4-значный SMS-код.
- `POST /api/v1/client/auth/verify-code` - проверить код и получить token.
- `GET /api/v1/client/profile` - профиль клиента.
- `GET /api/v1/services` - список активных услуг.
- `POST /api/v1/client/orders/checkout` - создать заявку.
- `GET /api/v1/client/orders` - история заявок клиента.
- `POST /api/v1/cleaner/auth/login` - вход клинера по телефону и 6-значному коду.
- `GET /api/v1/cleaner/profile` - профиль клинера.
- `GET /api/v1/cleaner/orders/available` - доступные заявки.
- `POST /api/v1/cleaner/orders/{order}/accept` - взять заявку.
- `POST /api/v1/cleaner/orders/{order}/start` - начать работу.
- `POST /api/v1/cleaner/orders/{order}/complete` - завершить заявку.
- `GET /api/v1/cleaner/orders/history` - история выполненных заявок.

## Документация API

Публичный контракт API ведется в `docs/api/openapi.yaml`. Человекочитаемые пояснения находятся в `docs/api/*.md`. Любое изменение маршрута, request body, response body, статуса ответа или схемы ошибки должно обновлять OpenAPI-контракт в том же изменении кода.

Документация внутри модулей допускается только для внутренней доменной логики, например workflow OTP или правила переходов статусов. Она не заменяет единый публичный API-контракт в `docs/api/openapi.yaml`.
